# Аудит проекта tg-ai-bot-teloxide — 2026-07-05

Полный аудит: SQL injection, prompt injection, security, документация vs код, Bot API 10.1.

## SQL Injection — ✅ Чисто

Все SQL-запросы параметризованы через `sqlx::query_as()` / `sqlx::query()` с биндингами `$1, $2, ...`.

- `db/telegram.rs` — все запросы параметризованы, включая `resolve_user_id` (username через `$1`)
- `features/stats/report.rs` — динамическая генерация SQL через `format!()` использует **только** `period.start_sql()`, который возвращает `&'static str` из `match` enum (не пользовательский ввод)
- `features/new_user_analysis.rs:1036-1158` — динамические имена колонок через `QueryBuilder`, но `audit_insert_columns()` возвращает хардкоженный `&'static [&'static str]` — безопасно, хотя хрупко (AP6)
- `features/voice/repo.rs`, `features/first_comment/repo.rs`, `features/memory/service.rs` — всё через биндинги

**Вердикт: SQL injection невозможен.**

## Prompt Injection — 🟡 Умеренный риск

### Что защищено хорошо

1. **Output validation** (`quality.rs`) — многослойная: длина, наличие ровно одного `{CHAT_LINK}`, whitelist labels, отсутствие raw links, отсутствие generic CTA, минимум кириллицы, запрет на английский-доминантный текст
2. **HTML rendering** (`render.rs`) — LLM output проходит через `normalize_ai_markers` → `strip_links` → `Html::text()` (экранирует `&`, `<`, `>`, `"`)
3. **`{CHAT_LINK}` replacement** — whitelist из 8 labels, URL жёстко из `config.chat_invite_url`
4. **Voice cleanup output** — весь текст через `Html::text()`

### M-PI1. Memory poisoning (Medium)
**`features/memory/service.rs:233-247`**

Пост канала конкатенируется в prompt как plain text (нет разделения system/user messages). Риск снижён: источник — только контролируемый канал `НедоNews`, output validator отбрасывает вредоносный текст, модель petit (90 токенов).

### M-PI2. Voice cleanup prompt injection (Low-Medium)
**`features/voice/cleanup.rs:69-92`**

ASR transcript конкатенируется в cleanup prompt. Злоумышленник через crafted voice может заставить cleanup-модель вернуть arbitrary JSON. Смягчается: JSON парсится строго, текст через `Html::text()`.

### M-PI3. Memory note injection (Low)
**`features/memory/service.rs:60-135`**

LLM генерирует заметку из поста, которая подмешивается в будущие промпты. Смягчается: только контролируемый канал, output validator.

## Security

### 🔴 S-1. Error response body может содержать секреты (High)
**`src/telegram/render.rs:190-211`**

`send_rich_message_request` логирует response body целиком в error message. Если API вернёт ошибку, тело ответа попадает в логи через `io_request_error`.

### 🟡 S-2. Gemini API key может просочиться в лог при ошибке (Medium)
**`src/llm/gemini.rs:49-61`**

`header("x-goog-api-key", self.api_key)` + `error_for_status()?`. Если tracing middleware логирует request headers при ошибке — утечка. reqwest по умолчанию не логирует, но кастомная instrumentation может.

### 🟡 S-3. TELOXIDE_TOKEN в URL для raw API calls (Medium)
**`src/user_profiles/service.rs:255`, `src/telegram/render.rs:181`**

Токен в URL для `getUserPersonalChatMessages` и `sendRichMessage`. Если HTTP client/log framework логирует URL при ошибке — утечка.

### ✅ HTML Injection — Чисто
Все отправляемые HTML через `Html::text()`, `Html::bold()`, `Html::link()` — экранируют `&`, `<`, `>`, `"`. `Html::raw_trusted()` только для внутренне сконструированного HTML. **Telegram ParseMode::Html injection невозможен.**

### 🟡 S-4. `normalize_ai_markers` неполная (Low)
**`src/text.rs:14-21`**

Заменяет только `—/–`, `«/»`, `Вот вариант:`, `Вариант:`. Не фильтрует markdown-заголовки `##`, bold `**`, bullet-листы, `---`.

### 🟡 S-5. HTTP client cache без eviction (Low)
**`src/http.rs:25-59`**

`HashMap<ClientKey, reqwest::Client>` растёт без bounds. На практике: мало timeout/proxy комбинаций, риск минимален.

### 🟡 S-6. Photo download без size limit в памяти (Medium)
**`src/features/first_comment/pipeline.rs:173-178`**

Всё фото → `Vec<u8>` + base64. Нет лимита на размер скачиваемого фото. Voice pipeline имеет лимит `VOICE_MAX_FILE_MB`, но first_comment — нет.

### 🟡 S-7. Concurrent profile refresh spawns без лимита (Low)
**`src/main.rs:128-154`**

Каждое сообщение → `tokio::spawn` для profile refresh (3 API-вызова). DB pool = 5 connections. Не жесткий DoS, но ненужная нагрузка.

## Документация vs код

### 🟡 D-1. README не перечисляет `/status`, `/topreact`, `/userstatus` (Medium)
В коде (`commands.rs`) есть:
- `/status` — alias для `/stats_day|week|month`
- `/topreact` — топ сообщений по реакциям
- `/userstatus` — alias для `/userstats`

В `README.md` и `docs/TECHNICAL.md` — нет.

### 🟡 D-2. TECHNICAL.md не упоминает `GEMINI_TTS_MODEL`, `PUBLIC_BASE_URL`, `STATIC_FILES_DIR` (Medium)
В коде (`config.rs`) эти поля есть и используются (gemini_tts — задел, public_base_url + static_files_dir — для rich-аватарок в stats).

### 🟡 D-3. `.env.example` содержит мёртвый `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID` (Low)
`cloudflare` нигде в `src/` не используется.

### 🟡 D-4. Voice timestamps — ближайший фикс, но не обновлён (Low)
`docs/TECHNICAL.md:353` — `TranscriptChapter.start_sec` хранится, но `render.rs` не выводит timestamp. Подтверждено кодом.

### 🟡 D-5. TECHNICAL.md не упоминает `sendRichMessage` (Bot API 10.1) (Medium)
Код имеет полную реализацию `InputRichMessage` / `sendRichMessage` (`render.rs:44-212`), помечена `#[allow(dead_code)]`.

### 🟡 D-6. Дефолт `LLM_TEMPERATURE` 0.35 в docs vs 0.45 в коде (Low)
**`docs/TECHNICAL.md:76`** — `LLM_TEMPERATURE=0.35`, **`src/config.rs:72`** — `env_f32("LLM_TEMPERATURE", 0.45)`.
