# NedoBot

**AI-слой для `НедоNews Chat`: первый комментарий под постом, память новостей, RAG, статистика и аккуратная автоматизация вокруг живого Telegram-чата.**

[Telegram-канал `НедоNews`](https://t.me/ned0news)

</div>

---

`NedoBot` — не универсальный бот-шаблон и не SaaS-заготовка. Это рабочая инфраструктура конкретного сообщества вокруг Telegram-канала [`НедоNews`](https://t.me/ned0news): бот смотрит на посты канала, понимает контекст чата, помогает запускать обсуждение, помнит прошлые новости и собирает метрики реакции аудитории.

Главная идея простая: **меньше ручной рутины, больше живого обсуждения**. Бот не заменяет людей в чате и не пытается изображать редакцию. Он берёт на себя повторяемые задачи: очистить пост от хвостов, передать контекст в LLM, написать первый комментарий, не повторить прошлые CTA и сохранить данные так, чтобы их можно было анализировать позже.

## Что это делает для чата

| Контур | Что происходит |
| --- | --- |
| Первый комментарий | Бот ловит авто-форвард поста из канала в discussion-чате и отвечает первым комментарием под ним. |
| Очистка поста | Служебный хвост вроде `Не теряем связь — VK / MAX` вырезается до отправки в модель. |
| Vision + LLM | Текст поста, главное фото, RAG, память и последние ответы бота уходят в vision-модель. |
| Память новостей | После комментария бот делает короткую заметку по посту и использует похожие заметки в будущих генерациях. |
| Антиповтор | Перед генерацией подтягиваются последние ответы, чтобы модель не зацикливалась на одинаковых заходах. |
| Статистика | Считаются сообщения, активные пользователи, реплаи, медиа, ссылки, реакции и активность после комментариев бота. |
| Архив событий | Входящие сообщения и raw Telegram JSON сохраняются в Postgres для будущего анализа и дообогащения. |

## Как выглядит основной сценарий

1. В канале выходит обычный редакционный пост.
2. Telegram автоматически форвардит его в привязанный discussion-чат.
3. Бот проверяет `source_channel`, авто-форвард и редакционный маркер.
4. Реклама и служебные публикации без маркера пропускаются.
5. Текст очищается от хвоста с внешними ссылками.
6. Самое большое фото поста отправляется в vision-модель.
7. В prompt добавляются RAG, релевантная память и последние ответы бота.
8. Модель возвращает комментарий с плейсхолдером `{CHAT_LINK}`.
9. Код сам заменяет плейсхолдер на HTML-ссылку и отправляет reply под постом.
10. Результат генерации, финальный HTML и дальнейшая реакция чата сохраняются в базе.

## Что уже есть

### Посты и комментарии

- распознавание авто-форвардов из канала через Telegram `forward_origin`;
- фильтр редакционных постов по маркеру;
- удаление VK/MAX-хвоста перед отправкой в LLM;
- vision-контекст по фото поста;
- генерация первого комментария через Ollama Cloud / Gemma vision model;
- HTML-ответ под постом с отключённым link preview;
- тематические custom/premium emoji для канала и отдельных техно-тем.

### Память и RAG

- короткие заметки по новостям после генерации комментария;
- keyword-based поиск похожих заметок;
- merge похожей памяти, чтобы не плодить дубли;
- tech RAG как страховка от устаревших утверждений;
- правило: **пост канала важнее RAG**, а RAG нужен не для пересказа, а для защиты от очевидной фактической кривизны.

### Метрики чата

- дневная, недельная и месячная статистика;
- сообщения, активные пользователи, реплаи, медиа и ссылки;
- посты канала и комментарии бота;
- реплаи на комментарии бота;
- reaction updates и reaction count updates;
- входы и выходы, если Telegram отдаёт `chat_member` updates;
- активность после комментария бота за 5 и 30 минут;
- расширенная карточка пользователя в `telegram_chat_users`.

## Внутренняя архитектура

```text
Telegram updates
      │
      ▼
teloxide dispatcher
      │
      ├── сохранение raw update/message в Postgres
      ├── определение авто-форварда из канала
      ├── фильтр редакционного поста
      ├── очистка текста и выбор фото
      │
      ▼
Prompt builder
      │
      ├── текст поста
      ├── vision-вход
      ├── tech RAG
      ├── похожая память
      ├── последние ответы бота
      │
      ▼
LLM / vision model
      │
      ▼
HTML renderer + Telegram reply
      │
      ▼
Postgres: jobs, generations, memory, stats
```

## Границы и ограничения

Это живой MVP, а не законченная платформа.

- Память пока keyword-based, без embeddings и pgvector.
- Merge заметок эвристический, качество памяти нужно иногда смотреть руками.
- Старые реакции Telegram Bot API не отдаёт задним числом.
- Join/leave и реакции зависят от того, какие update-типы Telegram реально присылает боту.
- Join-конверсия по отдельной invite-ссылке пока не считается автоматически.
- Админки пока нет: часть настроек живёт в `.env` и требует рестарта сервиса.
- Если LLM-провайдер падает или упирается в лимиты, job может потребовать ручного вмешательства.

## Ближайшие направления

- Расшифровка голосовых сообщений с очисткой ASR, пунктуацией, смысловыми таймкодами и fallback в файл для длинных сообщений.
- Более нормальная память через embeddings/pgvector вместо чистого пересечения keywords.
- Админка для настроек без рестарта сервиса.
- Отдельная аналитика конверсии invite-ссылок.
- Более строгий контур модерации: ссылки, новые пользователи, поведенческие сигналы, репутация.
- Удобные отчёты по динамике чата без ручных SQL-запросов.

## Зачем это всё

`НедоNews Chat` — это не просто комментарии под постами. Это место, где новости превращаются в обсуждение: кто-то приносит контекст, кто-то спорит, кто-то проверяет факты, кто-то докидывает мемы и технические детали.

`NedoBot` нужен как тихий инфраструктурный слой вокруг этого процесса. Он помогает начать разговор, не терять контекст, видеть реакцию аудитории и постепенно превращать хаотичный поток Telegram-сообщений в данные, с которыми можно работать.
